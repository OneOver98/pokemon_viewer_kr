# Implementation Plan: Headless PvPoke RL Environment

## Goal Description
Build a Reinforcement Learning (RL) training pipeline for Pokémon GO PvP using the open-source PvPoke battle simulation engine. The goal is to run the JS engine headlessly in Node.js, interface it with a Python-based RL training loop (using `stable-baselines3` and PyTorch), and parallelize the environments across CPU cores while utilizing the RTX 3060 GPU for the neural network training.

## User Review Required
No critical breaking changes, but please review the architectural choices (specifically how we bridge JS and Python via standard IO pipes).

## Proposed Architecture & Changes

### Node.js Headless Wrapper
We will write a Node.js script that reads and concatenates the essential PvPoke source files ([Pokemon.js](file:///c:/Users/jake/ai_1o/pvpoke/src/js/pokemon/Pokemon.js), [Battle.js](file:///c:/Users/jake/ai_1o/pvpoke/src/js/battle/Battle.js), [GameMaster.js](file:///c:/Users/jake/ai_1o/pvpoke/src/js/GameMaster.js), [ActionLogic.js](file:///c:/Users/jake/ai_1o/pvpoke/src/js/battle/actions/ActionLogic.js), etc.) and evaluates them within a Node environment or `vm` sandbox.
- **Data Loading:** The browser-based PvPoke uses `$.ajax` to load `data/gamemaster.min.json`. We will mock this or instantiate our own version of [GameMaster](file:///c:/Users/jake/ai_1o/pvpoke/src/js/GameMaster.js#170-186) by using Node's `fs.readFileSync` to load the JSON file locally.
- **Action/State Loop:** Instead of the `setInterval` loop used in [emulate()](file:///c:/Users/jake/ai_1o/pvpoke/src/js/battle/Battle.js#685-735) mode, we will expose an asynchronous or synchronous interface where the script waits for an action from standard input, calls `battle.step()`, and outputs the new state matrix, legal moves, and rewards to standard output.
- **Communication:** Python and Node.js will communicate via `stdin/stdout` using highly optimized JSON line protocols or simple struct packing.

### Python Gymnasium Environment
We will create a custom environment `PvPokeEnv` that inherits from Python's `gymnasium.Env`.
- **`reset()`:** Spawns/resets the Node child process or sends a reset command. Randomly samples Pokemon or requires them via configuration.
- **[step(action)](file:///c:/Users/jake/ai_1o/pvpoke/src/js/battle/Battle.js#269-624):** Sends the chosen action (e.g., Fast Move, Charged Move 1, Charged Move 2, Switch, Shield) to the Node child process, reads the parsing loop output, scales the rewards, and returns `observation`, `reward`, `terminated`, `truncated`, `info`.
- **GPU & Multiprocessing Scaling:**
  - We will use `stable-baselines3.common.vec_env.SubprocVecEnv` to spawn multiple `PvPokeEnv` instances. Each instance lives in its own Python worker process, which in turn holds its own single-threaded Node.js child process. This achieves parallelization across the CPU cores natively.
  - The PyTorch neural network built by `stable-baselines3` (PPO, DQN, or SAC) automatically uses CUDA/RTX 3060 for batching the observations from all sub-processes and optimizing the policy.

### Detailed Files to Create
#### [NEW] `ai_pipeline/pvpoke_env.js`
The headless sandbox wrapper script written in Node.js. Reads the PvPoke files into memory and acts as a standard IO server for the game loop.

#### [NEW] `ai_pipeline/pvpoke_gym.py`
The Python `gymnasium.Env` logic for starting/stopping the subprocess, formatting observations (e.g., current HP, energy, shields, buffs, swap timers), and interpreting actions.

#### [NEW] `ai_pipeline/train.py`
The main RL training loop using `stable-baselines3`. Wraps the `PvPokeEnv` with `SubprocVecEnv` (with a worker count equal to available CPU cores) and initiates the `PPO` or `DQN` training logic mapped to the `"cuda"` PyTorch device.

## Verification Plan
### Automated Tests
- Test Node Headless Runner: Ensure that issuing scripted commands directly through standard input correctly simulates turns, handles shields, tracks energy correctly, and ends the battle. We will write a small Python test suite for this (e.g., `pytest test_pvpoke_gym.py`) that steps through a known matchup.
- Test Multiprocessing Pipeline: Mock the RL agent taking random actions and verify that setting `num_envs=8` successfully scales the throughput of steps per second across CPU cores without crashes or deadlocks.

### Manual Verification
- Observe Tensorboard logs generated by `train.py` to ensure the mean episodic reward actually increases, validating that the integration architecture correctly aligns observations and actions.
